# StyleHub B2B — 5성능 개선 프로젝트

> 팀 프로젝트로 완성한 StyleHub B2B(패션 B2B 무역구매·배송대행 플랫폼)를 개인적으로 fork하여, k6 부하테스트 기반 성능 개선 작업을 진행하는 프로젝트입니다.

**시드 데이터 규모**

| Company | User | Product | Option | Address | CartItem | Order |
|---------|------|---------|--------|---------|----------|--------|
| 200     | 600  | 10,000  | 30,000 | 200     | 608      | 20,000 |

---

## 목표 기준치

시나리오별로 아래 기준을 k6 `thresholds`에 설정하고, 이를 기준으로 병목 여부와 개선 효과를 판단합니다.

| 시나리오     | 목표 p95 응답시간 | 목표 에러율 |
|----------|-------------|--------|
| 협의 목록 조회 | 300ms 미만    | 1% 미만  |
| 주문 생성    | 800ms 미만    | 1% 미만  |
| 결제 confirm | 1500ms 미만   | 1% 미만  |

조회는 엄격하게, 여러 테이블에 걸친 쓰기 작업인 주문 생성은 다소 여유 있게, 외부 API(토스) 호출이 포함된 결제는 가장 여유 있게 잡았습니다.

---

## 진행 상황

로컬 개발 환경 분리, Java Faker 기반 대량 시드 데이터(상품 1만 건, 주문 2만 건 등) 구축을 마쳤고, 우선순위 API를 대상으로 k6 부하테스트와 병목 개선을 진행 중입니다. 주문 생성 코드를 살펴보다 재고 미차감으로 인한 오버셀 버그를 발견해 수정하고 재검증까지 마쳤습니다. 결제 confirm은 실제 토스페이먼츠 `paymentKey`가 필요해 부하테스트가 막혀있었는데, 외부 API 호출 한 줄만 `@Profile` 기반 스텁으로 분리해 나머지 내부 로직(검증/DB 저장/주문 상태 전환)을 부하테스트할 수 있게 만들고 검증까지 마쳤습니다.

| 우선순위 시나리오  | 상태 |
|------------|---|
| 협의 목록 조회   | 부하테스트 완료, 목표 통과 (p95 85ms) |
| 주문 생성      | 부하테스트 완료, 목표 통과 (p95 235~370ms) |
| 결제 confirm | 부하테스트 완료, 목표 통과 (p95 330~710ms, VU 20, 외부 API 스텁 처리 기준) |

## 진행 내용

- 팀 공용 DB/Redis와 완전히 분리된 로컬 개발 환경 구성 (Docker)
- Java Faker(`datafaker`) 기반 시드 데이터 생성 스크립트 설계 및 구축, 도메인별 클래스로 리팩토링
- Hibernate 배치 insert 튜닝으로 대량 데이터 저장 성능 개선
- k6로 협의 목록 조회 부하테스트 완료 — 콜드 스타트, k6 쿠키 저장소 이슈 진단 및 해결
- 주문 생성 시나리오용 시드(ProductOption/Address/CartItem) 추가, 시드 데이터의 MOQ 검증 누락 버그 발견 및 수정
- k6로 주문 생성 부하테스트 완료 — 콜드 스타트 재확인, 목표 기준치 통과
- 재고 미차감으로 인한 오버셀 버그 발견 — 20명 동시 주문 시나리오로 재고 10개가 20개(200%) 팔리는 것을 재현
- `ProductOptionRepository`에 원자적 조건부 UPDATE(`decreaseStock`) 추가, `BuyerOrderService`에 재고 부족 시 주문 거부 로직 반영
- 같은 시나리오로 재검증 — 재고 10개만큼만 정상 판매되고 초과분은 거부되는 것을 SQL로 확인
- 결제 confirm의 유일한 외부 호출(`TossPaymentsClient.confirm`)만 `@Profile("loadtest")` 스텁으로 대체해, 나머지 내부 로직을 부하테스트로 검증 — VU 20에서 3회 반복 모두 목표 통과

## 원본 팀 프로젝트 배경

아래는 이 프로젝트의 원형이 된 팀 프로젝트(2026.06.02 ~ 2026.07.09, 5인)에 대한 설명입니다. 지금 진행 중인 성능 개선 작업은 이 팀 프로젝트가 끝난 뒤, 개인이 fork하여 이어가고 있는 것입니다.

- 일반 구매: 상품 조회 → 장바구니 → 주문 → 결제 → 배송 → 거래 확정
- 소싱 구매: 소싱 요청 → 견적 작성 및 협의 → 견적 채택 → 주문

### 팀 구성

| 이름  | 담당 도메인                                                      |
|-----|-------------------------------------------------------------|
| 조민재 | 견적, 협의(Negotiation), 전자계약, 장바구니/Checkout, 주문, 이의제기(Dispute) |
| 노현웅 | 인증/인가, 회원가입, 회사·직원 관리, 1:1 문의(WebSocket), 주소                |
| 김승민 | 상품(Product), 위시리스트                                          |
| 장백온 | 소싱(Sourcing), 알림/배송추적                                       |
| 김강민 | 결제(Payment), 정산(Settlement)                                 |
